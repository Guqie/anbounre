# Claude Skills 完整构建指南

## 一、Skills 是什么？

简单说，Skill 就是一个**文件夹**，里面装着一套教 Claude 做特定任务的说明书。好处是：你教一次，以后每次都能用，不用反复解释你的需求和偏好。

适合做成 Skill 的场景：按固定流程生成前端设计、写文档、做研报、跑多步骤工作流等重复性任务。

---

## 二、核心设计原则

**1. 渐进式披露（三层结构）**

- **第一层：YAML 头部** → 始终加载在系统提示中，让 Claude 知道"什么时候该用这个 Skill"
- **第二层：[SKILL.md](http://SKILL.md) 正文** → Claude 判断相关时才加载，包含完整指令
- **第三层：引用文件** → 按需查阅的补充资料（references/ 文件夹里的东西）

目的：省 token，按需加载。

**2. 可组合性** — 多个 Skill 可以同时加载，别假设自己是唯一的。

**3. 可移植性** — 同一个 Skill 在 [Claude.ai](http://Claude.ai)、Claude Code、API 上都能用。

---

## 三、Skills 与 MCP 的关系

用厨房打比方：

|MCP|Skills|
|---|---|
|专业厨房（工具、食材、设备）|菜谱（怎么做出好菜）|
|Claude **能做什么**|Claude **该怎么做**|

没有 Skill 时：用户连上 MCP 但不知道怎么用，每次从零开始，结果不稳定。 有了 Skill 后：预设工作流自动触发，结果一致，学习成本低。

## 四、规划与设计

### 4.1 从用例出发

动手写之前，先想清楚 2-3 个具体场景。好的用例定义包含：触发条件、执行步骤、最终结果。

### 4.2 三大常见类型

|类型|用途|典型例子|
|---|---|---|
|**文档/资产创建**|生成一致、高质量的输出|前端设计、docx/pptx/xlsx|
|**工作流自动化**|多步骤流程标准化|skill-creator 本身|
|**MCP 增强**|给 MCP 工具加上工作流指导|Sentry 代码审查|

### 4.3 定义成功标准

定量指标（参考值）：

- 90% 的相关查询能正确触发
- 完成工作流所需的工具调用次数
- 0 次失败的 API 调用

定性指标：

- 用户不需要手动提示下一步
- 工作流无需用户纠正即可完成
- 不同会话间结果一致

## 五、技术要求（重点！）

### 5.1 文件结构

```
your-skill-name/
├── SKILL.md          # 必须，主文件
├── scripts/          # 可选，可执行脚本
├── references/       # 可选，参考文档
└── assets/           # 可选，模板、字体等
```

### 5.2 硬性规则

- 文件**必须**叫 `SKILL.md`（大小写敏感，`skill.md` 不行）
- 文件夹名用 **kebab-case**：`notion-project-setup` ✅，不能有空格、下划线、大写
- **不要**在 Skill 文件夹里放 `README.md`（文档全写在 [SKILL.md](http://SKILL.md) 或 references/ 里）
- 名字不能包含 "claude" 或 "anthropic"（保留词）
- YAML 里**禁止** XML 尖括号 `< >`（防注入）

### 5.3 YAML 头部（最关键的部分）

最小格式：

```yaml
---
name: your-skill-name
description: 做什么 + 什么时候用。
---
```

**description 字段写法要点：**

- 必须同时包含"做什么"和"什么时候触发"
- 不超过 1024 字符
- 要写用户可能说的具体短语（如"帮我规划 sprint"）
- 如涉及文件类型要提到

好例子：`"管理 Linear 项目工作流…用户提到 'sprint'、'项目规划' 时触发"` 坏例子：`"帮助处理项目"` ← 太模糊，不会被触发

可选字段：`license`、`compatibility`、`metadata`（作者、版本、MCP 服务器名等）

### 5.4 正文写作建议

推荐结构：分步骤指令 → 示例场景 → 常见问题排查

写作原则：

- **具体可操作**：给出确切命令和预期输出，而不是"验证一下数据"
- **包含错误处理**：列出常见错误及解决方案
- **清晰引用资源**：明确告诉 Claude 去哪找参考文件
- **用渐进式披露**：核心指令放 [SKILL.md](http://SKILL.md)，详细文档放 references/

## 六、测试与迭代

### 6.1 三种测试方式

- **手动测试**（[Claude.ai](http://Claude.ai) 里直接跑）→ 快速迭代
- **脚本测试**（Claude Code 里自动化）→ 可重复验证
- **API 编程测试** → 系统化评估

**Pro Tip：先在一个难题上反复迭代，成功后再提取成 Skill。**

### 6.2 三类测试

**触发测试**：该触发时触发了吗？不该触发时没触发吧？

- 测试正面触发（直接说法、换种说法）
- 测试负面触发（无关话题不应触发）

**功能测试**：输出对不对？API 调用成功了吗？边界情况覆盖了吗？

**性能对比**：有 Skill vs 没 Skill，对话轮数、token 消耗、失败率的差异。

### 6.3 用 skill-creator 工具

内置于 [Claude.ai](http://Claude.ai)，可以帮你：生成 Skill 初稿、审查问题、建议测试用例、迭代改进。

用法：`"Use the skill-creator skill to help me build a skill for [你的用例]"`

### 6.4 迭代信号

|问题|信号|解决|
|---|---|---|
|触发不足|Skill 该加载时没加载|在 description 加更多关键词和触发短语|
|触发过度|无关查询也加载了|加"负面触发"说明，缩窄范围|
|执行问题|结果不一致、API 失败|改进指令，加错误处理|

## 七、分发与共享

### 7.1 当前安装方式

- 个人用户：下载 → 压缩 → 上传到 [Claude.ai](http://Claude.ai) 设置页面（或放到 Claude Code 目录）
- 组织管理员：可全工作区部署，集中管理

### 7.2 API 使用

通过 `/v1/skills` 端点管理，在 Messages API 中用 `container.skills` 参数添加。适合生产环境和自动化流水线。

### 7.3 推荐的发布流程

1. 在 GitHub 建公开仓库，写好 README（给人看的，不放进 Skill 文件夹）
2. 在 MCP 文档中链接到 Skill，说明搭配使用的价值
3. 写安装指南

### 7.4 宣传要点

- 聚焦**结果**而非功能：`"几秒钟搭建完整项目空间"` ✅ vs `"包含 YAML 头部的文件夹"` ❌
- 讲清 MCP + Skill 的协同故事

## 八、五大实战模式

|模式|适用场景|核心思路|
|---|---|---|
|**顺序工作流编排**|必须按特定顺序执行的多步流程|明确步骤顺序、步骤间依赖、每步验证、失败回滚|
|**多 MCP 协调**|跨多个服务的工作流|分阶段、阶段间传数据、进入下阶段前验证|
|**迭代精炼**|输出质量需要反复提升|初稿 → 质量检查 → 修改 → 再检查 → 达标交付|
|**上下文感知工具选择**|同一目标但不同情况用不同工具|决策树、备选方案、向用户解释选择原因|
|**领域专业知识**|需要嵌入专业判断（如合规检查）|先检查再执行、完整审计记录、清晰治理|

## 九、常见问题排查速查

|症状|原因|解决|
|---|---|---|
|上传失败 "Could not find [SKILL.md](http://SKILL.md)"|文件名大小写不对|必须是 `SKILL.md`|
|"Invalid frontmatter"|YAML 格式错（缺 `---` 分隔符、引号没闭合）|检查格式|
|Skill 不触发|description 太模糊|加具体触发短语|
|Skill 触发太频繁|description 太宽泛|加负面触发、缩窄范围|
|MCP 调用失败|连接/认证问题|先脱离 Skill 单独测试 MCP|
|指令没被遵循|指令太啰嗦/太含糊/关键指令被埋没|精简、把关键指令放最前面、考虑用脚本代替自然语言检查|
|响应变慢/质量下降|上下文太大|[SKILL.md](http://SKILL.md) 控制在 5000 字以内，详细内容移到 references/，减少同时启用的 Skill 数量|

## 十、上传前检查清单（精简版）

1. ☐ 确定了 2-3 个用例
2. ☐ 文件夹用 kebab-case 命名
3. ☐ 有 [SKILL.md](http://SKILL.md) 文件（拼写正确）
4. ☐ YAML 头部有 `--` 分隔符，name 和 description 都写了
5. ☐ description 包含"做什么"+"什么时候用"
6. ☐ 没有 XML 尖括号
7. ☐ 指令清晰可操作，有错误处理和示例
8. ☐ 测试了正面触发、负面触发、功能正确性
9. ☐ 压缩为 .zip 上传
10. ☐ 上传后在真实对话中测试并持续迭代
