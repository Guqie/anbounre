## 一、上下文窗口基础

### 1.1 什么是上下文窗口？

上下文窗口 (Context Window) 是 Claude Code 在单次会话中能够"记住"的最大 token 数量。当对话内容超出这个限制时，系统会自动进行上下文管理。

### 1.2 上下文限制机制

```
┌─────────────────────────────────────────────────────┐
│  会话开始 → 对话累积 → 接近上限 → 自动压缩/截断    │
└─────────────────────────────────────────────────────┘
```

**关键点：**
- 上下文包括：所有历史消息 + 工具调用记录 + 文件内容
- 超出限制后，最早的消息会被截断或摘要
- 主动管理比被动截断更有效

### 1.3 Token 计算参考

| 内容类型 | 约等于 Token 数 |
|---------|----------------|
| 1 个英文单词 | ~1.3 tokens |
| 1 个汉字 | ~1.5-2 tokens |
| 1 行代码 (平均) | ~10-20 tokens |
| 1KB 文本 | ~750 tokens |

## 二、核心会话管理命令

### 2.1 `/compact` - 上下文压缩

**用途：** 将当前会话的历史记录压缩成简洁的摘要，释放上下文空间。

**使用场景：**
- 长对话后上下文接近上限
- 完成一个任务阶段，准备开始新任务
- 需要保留关键信息但清除冗余对话

**命令格式：**
```bash
/compact [可选：自定义压缩指令]
```

**示例：**
```bash
# 基础用法
/compact

# 带指令压缩（推荐）
/compact 请保留所有代码变更和文件路径，摘要掉讨论过程

# 定向压缩
/compact 只保留最终结论，移除所有试错过程
```

**最佳实践：**
1. 在执行大型重构后使用 `/compact`
2. 压缩前明确告知 Claude 需要保留什么
3. 压缩后验证关键信息是否保留

---

### 2.2 `/save` - 保存会话快照

**用途：** 保存当前会话状态，包括上下文和进度。

**命令格式：**
```bash
/save [会话名称]
```

**示例：**
```bash
/save auth-module-refactoring
/save feature-user-authentication-v1
```

**注意事项：**
- 会话快照包含完整的上下文历史
- 适合保存重要的中间状态
- 定期保存防止意外丢失

---

### 2.3 `/resume` - 恢复会话

**用途：** 从之前保存的快照恢复会话。

**命令格式：**
```bash
/resume [会话名称]
```

**使用场景：**
- 继续之前未完成的任务
- 在不同设备间切换工作
- 恢复重要的调试会话

---

### 2.4 `/clear` - 清空上下文

**用途：** 清除当前会话的所有历史记录，从头开始。

**命令格式：**
```bash
/clear
```

**与 `/compact` 的区别：**
| 命令 | 效果 | 适用场景 |
|------|------|---------|
| `/compact` | 压缩保留摘要 | 需要保留关键信息 |
| `/clear` | 完全清空 | 开始全新任务 |

---

### 2.5 `/status` - 查看上下文状态

**用途：** 查看当前上下文使用情况和剩余容量。

**命令格式：**
```bash
/status
```

**输出信息：**
- 已用 token 数量
- 剩余上下文容量
- 当前会话时长

## 三、会话管理最佳实践

### 3.1 日常会话维护

**推荐工作流：**

```
开始任务 → 定期 /status 检查 → 接近 80% 时 /compact → 任务完成 /save → 新任务 /clear
```

### 3.2 压缩时机判断

| 上下文使用率 | 建议操作 |
|------------|---------|
| < 50% | 正常使用 |
| 50-70% | 考虑压缩 |
| 70-90% | 强烈建议压缩 |
| > 90% | 立即压缩或清空 |

### 3.3 有效的压缩指令模板

```bash
# 模板 1：代码开发
/compact 保留：所有最终代码版本、文件路径、API 变更。移除：调试过程、错误尝试、临时方案

# 模板 2：文档编写
/compact 保留：最终文档结构、关键决策、用户反馈。移除：草稿版本、讨论过程

# 模板 3：问题排查
/compact 保留：根本原因、解决方案、验证步骤。移除：排查过程、错误假设
```

### 3.4 会话命名规范

```bash
# 推荐格式：[类型]-[模块]-[日期]
/save feat-auth-20260228
/save fix-login-bug-20260228
/save refactor-api-v2-20260228

# 避免的命名
/save test1
/save 123
/save "asdfgh"
```

## 五、常见问题与解决方案

### Q1: 为什么 `/compact` 后 Claude 还是"忘记"了内容？

**原因分析：**
- 压缩指令不够明确
- 关键信息在摘要中被简化过度
- 上下文在压缩前已经超出限制

**解决方案：**
```bash
# 使用更具体的压缩指令
/compact 必须保留以下内容：
1. 所有 import 语句
2. 函数签名和返回值
3. 配置文件路径
4. 环境变量名称
```

### Q2: 如何避免在调试过程中上下文过快耗尽？

**预防策略：**
1. 使用 `/status` 定期检查
2. 每解决一个子问题就 `/compact`
3. 将已验证的解决方案写入文件而非留在对话中

### Q3: 会话保存后能在其他设备恢复吗？

**答案：** 取决于保存方式
- `/save` 保存的会话可以跨设备恢复（需同步）
- 建议配合云存储使用

### Q4: 自动压缩和手动压缩有什么区别？

| 特性 | 自动压缩 | 手动压缩 |
|------|---------|---------|
| 触发条件 | 系统判定 | 用户主动 |
| 压缩策略 | 通用算法 | 可定制 |
| 可控性 | 低 | 高 |
| 推荐度 | 不推荐依赖 | 强烈推荐 |

## 六、高级技巧与工作流

### 6.1 多任务并行管理

使用多个独立会话处理不同任务：

```
会话 A: 新功能开发 (保持活跃)
会话 B: Bug 修复 (按需恢复)
会话 C: 代码审查 (周期性使用)
```

### 6.2 会话模板库

创建常用会话模板：

```markdown
## 代码审查会话模板
/compact 前序审查意见已记录到 review-notes.md
当前审查重点：
1. 安全性检查
2. 性能影响评估
3. 代码规范符合性

## Bug 排查会话模板
问题描述：[简要描述]
影响范围：[模块/功能]
已尝试方案：[列表]
当前假设：[待验证]
```

### 6.3 会话审计与复盘

定期回顾保存的会话：

```bash
# 创建会话日志
ls -la .claude/sessions/ > session-log.md

# 分析高频操作
grep "/compact" session-log.md | wc -l
```

### 6.4 自动化脚本

```bash
#!/bin/bash
# 会话管理自动化脚本

# 每日会话清理
daily_cleanup() {
  echo "Cleaning up old sessions..."
  find .claude/sessions -name "*.json" -mtime +7 -delete
}

# 会话备份
backup_sessions() {
  tar -czf sessions-backup-$(date +%Y%m%d).tar.gz .claude/sessions/
}
```
